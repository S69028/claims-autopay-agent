# AGENTS.md

## 프로젝트 목적
이 프로젝트는 보험금 지급심사 워크플로우에서 **`자동심사 현황분석 Agent`**를 기획·검증하기 위한 것이다.

이 Agent의 목표는 다음과 같다.
- 자동지급 현황을 운영관리자 관점에서 빠르게 읽을 수 있게 한다.
- 월말 스냅샷 기준의 전월 대비 변화와 segment 변화를 설명한다.
- 자동지급률, 처리효율, 안정 상태를 상시 노출해 운영 판단과 상위 보고를 돕는다.
- 월간 리포트를 자동 생성·자동 발송하고, 대시보드에서 재조회할 수 있게 한다.

## 핵심 전제
- 1차 사용자는 **운영관리자**다.
- `심사 Auditor`와 `심사담당자`는 향후 확대 대상이다.
- 대시보드는 `운영모니터링`과 `리포트보관함`으로 나눈다.
- 조회기간은 **반기 기준**을 기본값으로 두되, 필터로 변경 가능하게 한다.
- 성과는 **판단값이 아니라 ROI 보고용**이다.
- 성과 해석은 **운영효율 중심**으로 한다.
- `자동지급 대상군`의 월말 스냅샷 변화를 기준으로 본다.
- `비율 변화`를 우선 해석하고, 필요 시 `비용환산 기준`으로 설명한다.
- 변화가 없어도 지표는 계속 노출하고, 상태는 `안정`으로 명시한다.

## 현재 스코프
### 만드는 것
- 운영모니터링 화면
- 리포트보관함
- 월간 리포트 자동 생성
- 월간 리포트 자동 발송
- 월간 리포트 재다운로드
- 조회기간 필터
- 자동지급률 그래프
- 성과 표시
- segment 변화 설명
- 월말 기준 전월 대비 변화 설명
- 안정 상태 표시

### 만들지 않는 것
- 청구유형 분류
- 최종 지급 여부 확정
- 지급금액 단독 확정
- 사람 심사가 필요한 건을 자동지급으로 강하게 밀어주는 기능
- 보조 입력만으로 자동 확정하는 기능
- 외부 API 기반의 민감정보 직접 전송

## 출력 원칙
- 추천은 하되, 최종 확정은 하지 않는다.
- 판단이 애매하면 신뢰도를 낮춘다.
- 보조 입력은 판단을 바꾸기보다 신뢰도를 조정하는 데 사용한다.
- `운영관리자`가 바로 이해할 수 있도록 설명 가능성을 우선한다.
- `월말 기준 전월 대비 변화`가 핵심이므로, 시계열 해석이 가능해야 한다.

## 설계 원칙
- factor별로 서로 다른 기준선과 가중치를 둘 수 있다.
- 기준선은 감이 아니라 데이터와 운영 판단을 바탕으로 조정한다.
- 운영 중 조정 가능한 구조로 둔다.
- 가능한 한 설명 가능성을 유지한다.
- `안정` 상태는 단순히 미변화가 아니라, 화면에서 명시적으로 보여준다.
- `segment 변화`는 변화 사실만이 아니라 어떤 변화가 있었는지까지 설명한다.

## 판단 우선순위
이 Agent를 검증할 때는 다음 순서로 본다.
1. 운영관리자가 실제로 쓰는가
2. 월말 스냅샷 기준의 변화와 segment 변화를 설명할 수 있는가
3. ROI 보고용 성과가 운영효율 중심으로 해석되는가
4. 안정 상태와 변화 상태를 일관되게 보여주는가
5. 주기적 조정이 가능한가

## 대화 규칙
- 먼저 가정과 빈칸을 확인하는 질문부터 한다.
- 한 번에 많은 결론을 내리지 않는다.
- 사용자가 `정리해줘`라고 하기 전에는 결론을 쓰지 않는다.
- 추측은 추측이라고 표시한다.
- 일반론보다 지급심사 운영과 직접 연결되는 질문과 판단만 한다.

## 검증 포인트
다음 항목을 우선 검증한다.
- 전월 대비 비율 변화가 실제로 운영 판단에 도움이 되는지
- 자동지급 대상군 기준이 일관된지
- segment 변화 설명이 충분히 구체적인지
- ROI 보고용 성과 산식이 조직 내 해석과 맞는지
- 안정 상태 표시가 정보 과잉 없이 유효한지
- 월말 스냅샷 기준만으로도 충분한지

## 확인해야 할 데이터
가능하면 아래 데이터를 확보한다.
- 월말 스냅샷 지급데이터
- 자동지급 대상군 정의와 월별 추이
- segment 변경 이력
- 전월 대비 증감률 이상징후 사례
- 성과 산식과 비용환산 기준
- 운영관리자가 실제 상위보고에 쓰는 보고서 샘플
- 안정 상태로 간주되는 기준

## 검증질문
이 Agent의 설계와 운영 가능성을 검증하기 위해 아래 질문을 우선 확인한다.

### 가장 약한 가정
- 운영관리자가 대시보드와 월간 리포트를 실제로 사용하게 되는가?
- 월말 기준 전월 대비 변화만으로도 운영 상황을 설명할 수 있는가?
- segment 변화 설명이 운영관리자와 상위보고자 모두에게 충분히 이해되는가?

### 데이터 부족
- 월말 스냅샷 기준 지급데이터가 충분히 있는가?
- 자동지급 대상군과 segment 변화 이력을 볼 수 있는가?
- 비용환산 기준과 성과 산식이 문서화되어 있는가?
- 안정 상태의 기준을 결정할 운영 규칙이 있는가?

### 운영 위험
- 운영관리자가 보는 화면이 너무 복잡해지지 않는가?
- 변화 없는 지표를 계속 노출했을 때 정보 과잉이 생기지 않는가?
- 월말 스냅샷이 흔들릴 때 잘못된 보고가 나가지 않는가?
- segment 변화 설명이 추정에 머물지 않고 근거를 유지하는가?

### 사람 승인 지점
- 리포트 기준이나 임계치를 누가 승인하는가?
- 안정 상태 정의를 누가 바꾸는가?
- 성과 산식과 비용환산 기준을 누가 확정하는가?
- 운영관리자 외 사용자로 확장할 때 누구의 승인이 필요한가?

## 기본 답변 톤
- 기획 보고서처럼 구조화하되 실행 중심으로 쓴다.
- 긴 설명보다 표, 우선순위, 결정 기준, 다음 액션을 우선한다.
- 확실한 사실과 추정을 구분한다.
- 데이터가 부족하면 부족하다고 말한다.
