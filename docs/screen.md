# Screen — 운영모니터링 / 리포트보관함 1차

> 6단계 `운영모니터링 화면 1차`의 기준 문서다.  
> 현재 HTML preview와 Figma 파일에 맞춰, 실제 구현할 화면 구조와 정보 우선순위만 고정한다.

## 1) 목적

- 운영관리자가 월말 기준 자동지급 현황을 가장 먼저 읽는 첫 화면을 정의한다.
- 숫자, 상태, 변화 설명, 리포트 진입점을 한 화면에서 연결한다.
- `운영모니터링`과 `리포트보관함`을 같은 앱 안의 두 개 view로 본다.

## 2) 디자인 방향

- enterprise analytics dashboard
- premium B2B SaaS console
- calm, trustworthy, data-dense, structured
- dark navy shell + light content canvas
- restrained blue accents
- crisp tables, compact KPI cards, minimal charts

## 3) 현재 화면 구조

### 3-1. 좌측 내비게이션

**구성**
- 운영모니터링
- 리포트보관함

**규칙**
- 현재 위치는 active 상태로 강조한다.
- 운영모니터링이 기본 진입점이다.
- 월간 리포트와 설정은 1차 화면의 독립 탭으로 두지 않는다.

### 3-2. 운영모니터링 view

#### 상단 헤더
**표시 항목**
- 화면명: `자동지급 운영 모니터링`
- 설명 문구
- 요약 pill: 기준월
- 요약 pill: 조회기간

**규칙**
- 헤더에는 `데이터 확정 여부`를 별도 배지로 노출하지 않는다.
- 확정월만 공식 기준으로 사용하지만, 화면 상단은 기준월과 조회기간 중심으로 단순하게 둔다.
- 필터와 요약 pill이 중복되지 않도록, pill은 현재 값 요약, 아래 row는 조작용 필터로 둔다.

#### 필터 row
**표시 항목**
- 조회기간
- 기준월
- 상태
- 검색

**규칙**
- 기본 조회기간은 최근 6개월이다.
- 그래프는 6개월 단위로 확대/축소할 수 있어야 한다.
- 상태와 검색은 보조 필터다.

#### KPI 카드 영역
**카드 항목**
- 자동지급률
- 처리효율
- 자동지급 건수
- 제외/인심사 건수
- 상태 라벨

**표시 규칙**
- 숫자를 가장 크게 보여준다.
- 라벨은 숫자 아래에 붙인다.
- 상태 라벨은 `안정`, `변화`, `미확정` 중 하나다.
- `확인 필요`는 상태가 아니라 보조 문구로 쓴다.

#### 자동지급 추이 영역
**표시 항목**
- 6개월 동안의 보험금 청구건수와 자동지급률 듀얼축 그래프

**규칙**
- 가로축은 청구년월이다.
- 좌측 축은 청구건수, 우측 축은 자동지급률이다.
- 범례는 그래프 하단 중앙에 별도 영역으로 배치한다.
- 그래프는 추세 읽기용이며, 상세 해석은 아래 패널에서 보완한다.

#### 성과 계산 영역
**표시 항목**
- 청구년월 선택
- 적용단가 입력
- 계산 버튼
- 계산 결과: 청구건수, 자동지급률, 성과

**규칙**
- 성과는 ROI 보고용으로 보여준다.
- 계산 결과는 표 형태로 짧게 노출한다.
- 선택 계산임을 배지로 표시한다.

#### 비율 카드 영역
**구성**
- 접수채널별 비율
- 진료구분별 비율

**규칙**
- 접수채널별 비율은 가로 막대형으로 보여준다.
- 진료구분별 비율은 파이차트로 보여준다.
- 두 카드는 성과 계산 옆 또는 바로 아래에서 촘촘하게 배치한다.

#### 변화 요약 영역
**표시 항목**
- 전월 대비 자동지급률 변화
- 전월 대비 처리효율 변화
- 자동지급 유형 상위 변화
- segment 변화 표

**규칙**
- 변화가 없어도 항목은 숨기지 않는다.
- segment 변화는 변경 전 / 변경 후 / 건수 / 비중 / 변화 사유로 표기한다.
- 변화가 증가인지 감소인지가 바로 읽히게 표시한다.

#### 자동지급 적용현황 영역
**표시 항목**
- 자동지급 유형
- 대상건수
- 전체 자동지급건수 대비 비율
- 적용여부 상태 배지

**규칙**
- `On / Off`는 실제 제어 버튼이 아니라 상태 배지다.
- 상위 5개 유형을 우선 보여준다.

### 3-3. 리포트보관함 view

**표시 항목**
- 리포트 목록 테이블
- 구독계정 관리 테이블
- 특정년월 선택 + 발송테스트 버튼

**리포트 목록 컬럼**
- 기준월
- 생성일
- 발송일
- 리포트 상태
- 파일명
- 액션

**리포트 상태**
- 생성됨
- 발송됨
- 미발송
- 재생성 필요

**구독계정 관리 컬럼**
- 이메일 주소
- 등록년월
- 상태
- 액션

**규칙**
- 리포트보관함은 재조회와 재다운로드 중심이다.
- 월말 스냅샷이 확정된 리포트만 공식 생성 대상으로 본다.
- 보관함의 상태는 생성/발송/재생성 필요 중심으로 관리한다.
- PoC 단계에서는 특정년월을 선택해 수동 발송테스트를 실행할 수 있게 한다.

## 4) 정보 우선순위

1. 기준월과 핵심 KPI
2. 자동지급 추이
3. 성과 계산
4. 비율 카드
5. 변화 요약
6. 자동지급 적용현황
7. 리포트보관함 목록

## 5) 표시 문구 원칙

- 숫자는 짧게, 해석은 한 줄로 쓴다.
- 상태 문구는 장황하지 않게 쓴다.
- 운영자가 바로 읽을 수 있도록 전문 용어는 최소화한다.
- 상단 헤더와 필터는 중복되지 않게 역할을 나눈다.

## 6) 예외 처리

- 미확정이면 KPI와 그래프는 보이되, 상태는 `미확정`으로 표시한다.
- 데이터가 없으면 `0` 또는 `-`를 임의로 남기지 말고 규칙적으로 표현한다.
- 필터 결과가 없으면 빈 화면 대신 안내 문구를 보여준다.
- 리포트보관함에서는 확정월이 아니면 공식 리포트를 만들지 않는다.

## 7) 구현 체크포인트

- 운영모니터링과 리포트보관함이 같은 앱 안의 두 view로 자연스럽게 전환되는가
- 자동지급 추이의 축/범례 배치가 겹치지 않는가
- 성과 계산과 비율 카드가 한 화면에서 밀도 있게 읽히는가
- 변화 요약에서 segment 변경 전후가 충분히 구체적으로 보이는가
- archive view가 단순 파일 목록이 아니라 재조회와 구독 관리 화면으로 읽히는가
