# Progress

## 바이브 코딩 단계

진행 방식:
- 각 단계는 한 목적만 가진다.
- 단계가 끝날 때마다 완료 여부를 기록한다.
- 다음 단계로 넘어가기 전에는 반드시 승인을 받는다.
- 초급자도 따라갈 수 있도록 구현 범위를 작게 유지한다.

| 단계 | 목적 | 상태 | 완료 기준 |
|---|---|---|---|
| 1 | 요구사항 고정 | 완료 | 사용자, 범위, 제외 범위, 핵심 원칙이 한 장으로 정리됨 |
| 2 | 데이터 계약 정리 | 완료 | 입력 테이블 구조, 컬럼 정의, CREATE TABLE DDL, 공통 용어가 정리됨 |
| 3 | 기능명세 구체화 | 완료 | 화면, 문구, 지표, 상태 표시, 리포트 항목이 문장 단위로 정리됨 |
| 4 | 지표 계산 분리 | 완료 | 자동지급률, 전월 대비 변화, segment 변화, 성과 계산이 함수로 분리됨 |
| 5 | 상태 판정 분리 | 완료 | `안정`, `변화`, `미확정` 규칙이 분리됨 |
| 6 | 운영모니터링 화면 1차 | 완료 | 핵심 지표가 한 화면에 표시됨 |
| 7 | 데이터 시뮬레이션 | 완료 | dummy data를 DB에 적재해 대시보드와 리포트 흐름을 시뮬레이션함 |
| 8 | 리포트 생성 분리 | 완료 | 월간 리포트가 자동 생성됨 |
| 9 | 자동 발송 / 보관 | 완료 | 리포트가 발송되고 보관함에 저장됨 |
| 10 | 예외 처리 설계 | 보류 | 데이터 누락, 미확정, 발송 실패에 대한 처리 기준이 있음 |
| 11 | 테스트 고정 | 완료 | 핵심 계산과 상태 판정이 테스트로 보호됨 |
| 12 | 검증과 반복 개선 | 대기 | 운영 피드백을 반영해 기준을 조정할 수 있음 |

## 1단계 산출물 — 요구사항 고정

### 문제·사용자(ICP)
- 1차 사용자는 보험금 자동지급 운영 현황을 월말 기준으로 읽고 보고해야 하는 `운영관리자`다.
- 향후 확대 대상은 `심사 Auditor`, `심사담당자`다.
- 이들은 자동지급 대상군의 변화, segment 변화, 자동지급률, 처리효율, 안정 상태를 빠르게 이해해야 한다.

### JTBD
- 월말 기준 자동지급 운영 현황을 빠르게 파악하고, 전월 대비 변화와 segment 변화를 설명한다.
- ROI 보고용 성과를 운영효율 중심으로 읽는다.
- 변화가 없어도 `안정` 상태를 명시적으로 확인한다.

### 범위
#### 만드는 것
- 운영모니터링 화면
- 리포트보관함
- 월간 리포트 자동 생성
- 월간 리포트 자동 발송
- 월간 리포트 재다운로드
- 조회기간 필터
- 자동지급률 그래프
- 성과 표시
- segment 변화 설명
- 월말 기준 전월 대비 변화 설명
- 안정 상태 표시

#### 만들지 않는 것
- 청구유형 분류
- 최종 지급 여부 확정
- 지급금액 단독 확정
- 사람 심사가 필요한 건을 자동지급으로 강하게 밀어주는 기능
- 보조 입력만으로 자동 확정하는 기능
- 외부 API 기반의 민감정보 직접 전송

### 핵심 원칙
- 추천은 하되, 최종 확정은 하지 않는다.
- 판단이 애매하면 신뢰도를 낮춘다.
- 보조 입력은 판단을 바꾸기보다 신뢰도를 조정하는 데 사용한다.
- 운영관리자가 바로 이해할 수 있도록 설명 가능성을 우선한다.
- 월말 기준 전월 대비 변화가 핵심이므로 시계열 해석이 가능해야 한다.

### 완료 기준
- 위 사용자, JTBD, 범위, 제외 범위, 핵심 원칙이 한 장에서 같은 의미로 읽힌다.
- 다음 단계인 `데이터 계약 정리`로 넘어가기 위한 기준이 흔들리지 않는다.

## 단계 운영 원칙

- `1단계`는 요구사항과 범위를 고정한다.
- `2단계`는 입력 데이터와 공통 용어를 고정한다.
- `3단계`는 기능명세를 문장 단위로 구체화한다.

## 2단계 산출물 — 데이터 계약 정리

현재 이 단계는 `더미 데이터 정의`가 아니라, 실제 개발에 쓸 `테이블 구조`, `컬럼 정의`, `CREATE TABLE DDL`을 고정하는 단계로 다시 정리한다.

### 공통 용어
| 용어 | 의미 | 비고 |
|---|---|---|
| 지급데이터 | 보험금 지급 관련 원천 데이터 | 실제 테이블 구조 기준으로 정의 필요 |
| 월말 스냅샷 | 월말 기준으로 고정한 대상군 상태 | 전월 대비 비교의 기준 시점 |
| 자동지급 대상군 | 자동지급 현황을 계산할 분모 집합 | 비율 계산의 기준 |
| segment | 대상군을 나누는 운영 구분 값 | 변화 설명의 핵심 축 |
| segment 변경 이력 | segment가 언제 어떻게 바뀌었는지 남긴 기록 | 변화 원인 설명의 근거 |
| 자동지급률 | 자동지급 처리된 비율 | 운영모니터링 핵심 지표 |
| 처리효율 | 운영효율을 보여주는 성과 지표 | ROI 보고용으로 사용 |
| 안정 상태 | 변화가 없거나 허용 범위 내인 상태 | 화면에 명시적으로 노출 |
| 미확정 상태 | 월말 데이터가 아직 확정되지 않은 상태 | 최종 리포트 생성 보류 기준 |

### 입력 데이터 계약
| 데이터 | 필수 여부 | 사용처 | 메모 |
|---|---|---|---|
| 지급데이터 | 필수 | 자동지급률, 성과, 리포트 | 원천 값 |
| 월말 스냅샷 | 필수 | 전월 대비 변화, 상태 판정 | 비교 기준 |
| 자동지급 대상군 | 필수 | 비율 계산 | 분모 역할 |
| segment 변경 이력 | 필수 | 변화 설명 | 설명 근거 |
| 비용환산 기준/단가 | 필수 | 처리효율, 성과 | ROI 보고용 |
| 조회기간 필터 값 | 선택 | 화면 조회 | 기본값은 반기 |
| 월별 자동지급 현황/리포트 이력 | 선택 | 보관함, 재조회 | 보관함 조회용 이력 |

### 출력 데이터 계약
| 데이터 | 생성 주체 | 사용처 | 메모 |
|---|---|---|---|
| 전월 대비 변화 요약 | 계산 로직 | 화면, 리포트 | 비율 변화 우선 |
| segment 변화 설명 | 설명 로직 | 화면, 리포트 | 변경 전/후 차이 포함 |
| 자동지급률 | 계산 로직 | 화면 | 핵심 지표 |
| 처리효율/성과 | 계산 로직 | 화면, 리포트 | ROI 보고용 |
| 상태 라벨 | 판정 로직 | 화면, 리포트 | `안정`, `변화`, `미확정` |

### 계약 원칙
- 입력과 출력을 분리해서 정의한다.
- 숫자 계산은 4단계에서 다루고, 여기서는 데이터 이름과 의미, 테이블 구조를 고정한다.
- 용어가 바뀌면 먼저 이 표를 수정한다.

### 추가로 필요한 초안
- 지급데이터 조회 테이블
- 자동지급유형 정의 테이블
- 자동지급제외유형 정의 테이블
- 각 테이블의 컬럼명, 타입, 키, null 허용 여부, 설명
- 각 테이블의 CREATE TABLE 문
- 테이블 간 연결 관계
- 월말 스냅샷의 기준 컬럼
- segment 변경 이력의 기준 컬럼
- 상태 판정에 필요한 최소 컬럼

### 현재 확인된 입력
- 사용자가 지급데이터 조회 테이블 초안을 보유하고 있다.
- 사용자가 자동지급유형과 자동지급제외유형 정의 초안을 보유하고 있다.
- 위 초안이 있어야 2단계를 완결할 수 있다.

### 완료 기준
- 화면과 리포트에서 쓰는 공통 용어가 하나로 정리된다.
- 입력 테이블 구조와 컬럼 정의가 확정된다.
- CREATE TABLE DDL이 확정된다.
- 출력 데이터와 상태 판정에 필요한 최소 필드가 확정된다.
- 3단계 기능명세를 작성할 때 혼동할 용어가 남지 않는다.

## 3단계 산출물 — 기능명세 구체화

이 단계에서는 화면과 리포트의 **보이는 형태**를 먼저 고정한다.  
구현 방식보다 `무엇을, 어떤 순서로, 어떤 문구로 보여줄지`를 문장 단위로 정리한다.

### 꼭 정할 것
- 운영모니터링 첫 화면의 고정 영역
- 자동지급률과 처리효율의 표시 방식
- 전월 대비 변화 설명 문구
- segment 변화 설명 문구
- 안정 / 변화 / 미확정 상태 문구
- 조회기간 필터 기본값과 변경 가능 범위
- 리포트보관함의 목록 항목과 재조회 동작
- 월간 리포트 본문에 반드시 들어갈 항목

### 기능명세 기준
- 한 기능마다 `보여줄 값`, `보여줄 위치`, `기본 상태`, `예외 상태`를 적는다.
- 숫자 지표와 설명 문구를 분리해서 쓴다.
- 사용자 화면과 월간 리포트가 같은 근거를 보도록 맞춘다.
- 불확실한 값은 `미확정` 또는 `확인 필요`로 남긴다.

## 4단계 산출물 — 지표 계산 분리

이 단계에서는 화면/문서와 별개로, 숫자를 계산하는 기준을 고정한다.  
계산 규칙의 단일 기준 문서는 `docs/metrics.md`다.

### 꼭 정할 것
- 자동지급률 계산식
- 전월 대비 변화 계산식
- segment 변화 계산식
- 처리효율 계산식
- 각 지표의 분모/분자
- 분모 0일 때의 처리 방식
- 첫 달 또는 전월 데이터 부재 시 처리 방식

### 완료 기준
- 운영모니터링과 월간 리포트가 같은 숫자를 보게 된다.
- 지표 계산식이 `docs/metrics.md`에 정리된다.
- 각 지표를 함수 단위로 분리할 수 있는 기준이 생긴다.

## 5단계 산출물 — 상태 판정 분리

이 단계에서는 숫자 계산 결과를 바탕으로 `안정`, `변화`, `미확정` 상태를 어떻게 정할지 고정한다.  
상태 판정 기준의 단일 문서는 `docs/status.md`다.

### 꼭 정할 것

- `미확정` 판정 조건
- `안정` 판정 조건
- `변화` 판정 조건
- 기준월 확정 여부 반영 방식
- 전월 비교 불가 시 처리 방식
- 임계치 조정 가능 여부

### 완료 기준
