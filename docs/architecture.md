# architecture.md — 입력 → 처리 → 출력 〔D5 핵심〕

> `자동심사 현황분석 Agent`의 입력, 처리, 출력, 사람 승인 지점, 실패 시 동작을 한 장으로 정리한 구조 문서.

| 섹션 | 내용 |
|---|---|
| Goal | 운영관리자가 자동지급 현황의 월말 변화, 청구 특성 변화, 자동지급률, 처리효율, 안정 상태를 빠르게 읽고 상위 보고할 수 있게 한다. |
| Input | 지급데이터 · 자동지급 대상군 월말 스냅샷 · 청구 특성 요소(receipt_channel, treatment_type, hospital_type, diagnosis_code_primary) · 지급데이터의 segment_before/after/reason(원본 기록) · 자동지급률 데이터 · 비용환산 기준/단가 · 월별 자동지급 현황/리포트 이력 · 조회기간 필터 값 |
| Process | ① 월말 스냅샷 수집 → ② 전월 대비 비율 변화 계산(자동지급 대상군 기준) → ③ 청구 특성 변화 분석 및 설명 생성 → ④ 처리효율/성과 계산(ROI 보고용) → ⑤ 안정/변화/미확정 상태 판정 → ⑥ 운영모니터링 화면 갱신 → ⑦ 월간 리포트 자동 생성(확정월만) → ⑧ 매월 1일 자동 발송 → ⑨ 리포트보관함 누적 저장 |
| Tools | 데이터 저장소(지급데이터/이력) · 리포트 생성 모듈 · 발송 모듈 · 운영모니터링 화면 · 리포트보관함 · 조회 필터 · 상태 판정 규칙 |
| Output | 운영모니터링 화면(자동지급률·처리효율·상태 라벨·전월 대비 변화·청구 특성 요약·추이 그래프) · 자동지급률 그래프 · 성과 표시 · 청구 특성 변화 설명 · 안정/변화 상태 라벨 · 월간 리포트 파일(기준월·비교월·성과·상태·요약 문구·발송 이력·본문 섹션·타이틀 구조·필요 시 시각화 포함) · 자동발송 결과 · 리포트 재다운로드 · 리포트보관함 목록 |
| HITL | 운영관리자가 월간 리포트와 대시보드 해석을 최종 확인한다. 안정 상태 정의와 임계치 조정은 운영 거버넌스 판단에 따른다. 청구 특성 변화 해석 기준은 운영정책에 따라 보완한다. |
| Failure | 월말 데이터 미확정 → `미확정` 상태로 표시하고 최종 리포트 생성 보류 / 분모 정의 불일치 → 비율 계산 중단 후 확인 필요 / 청구 특성 원천값 누락 → 변화 설명은 생성하되 `확인 필요`로 표기 / 발송 실패 → 재발송 큐에 남기고 보관함에는 생성본만 저장 |
| Cost | 월 1회 리포트 생성 + 운영자 조회 중심. 정량 비용은 아직 확인 필요. 보고서 생성은 배치형, 조회는 경량 계산으로 설계한다. |

> 실제 구현에서는 지급데이터와 월말 스냅샷을 분리된 테이블과 조회 조건으로 유지한다.

### 데이터 계약 메모
- 입력은 `지급데이터`, `월말 스냅샷`, `자동지급 대상군`, `지급데이터의 segment_before/after/reason`, `비용환산 기준/단가`, `조회기간 필터 값`으로 본다.
- 출력은 `전월 대비 변화 요약`, `청구 특성 변화 설명`, `자동지급률`, `처리효율/성과`, `상태 라벨`로 본다.
- `안정 상태`는 상태 라벨 중 하나이며, 단순 미변화가 아니라 허용 범위 내 상태를 뜻한다.
- `데이터 확정 여부`는 해당 월의 월말 스냅샷이 공식 기준으로 확정되었는지를 뜻한다.
- 기본 화면은 가장 최근에 확정된 월말 스냅샷을 기준으로 보여주고, 미확정 월은 공식 비교 기준과 리포트 생성 대상에서 제외한다.
- 그래프 조회는 6개월 단위로 확대/축소할 수 있게 한다.

## To-be Workflow

1. **데이터 수집**
   - 지급데이터와 자동지급 대상군 월말 스냅샷을 적재한다.
   - 지급데이터의 segment_before / segment_after / segment_change_reason는 원본 기록으로 유지하고, 청구 특성 요소와 비용환산 기준을 함께 연결한다.

2. **월말 비교**
   - 월말 기준으로 전월 대비 변화를 계산한다.
   - `비율 변화`를 우선 계산하고, 기준 모수는 `자동지급 대상군`으로 둔다.

3. **변화 설명 생성**
   - 청구 특성 요소가 전월 대비 자동지급률에 어떤 영향을 줬는지 설명 문장을 만든다.
   - 변화 전/후 차이 대신 factor별 전월 대비 비율 차이를 함께 보여준다.
   - 원인 정보가 부족하면 `확인 필요`로 남긴다.

4. **성과/안정 판정**
   - ROI 보고용 성과를 계산한다.
   - 운영효율 중심으로 표시한다.
   - 변화가 없거나 허용 범위 내면 `안정` 상태로 명시한다.

5. **운영모니터링 반영**
   - 대시보드 첫 화면에서 `그래프 / 자동지급률 / 성과 / 안정 상태`를 고정 노출한다.
   - 사용자는 기본적으로 운영모니터링 화면에서 시작한다.
   - 필요 시 기간 필터로 반기 외 구간도 조회한다.

6. **월간 리포트 생성**
   - 대시보드에서 사용한 근거를 바탕으로 월간 리포트를 자동 생성한다.
   - 리포트는 기준월, 비교월, 자동지급률, 전월 대비 변화, 청구 특성 변화, 처리효율/성과, 안정 상태, 짧은 해석 문구를 포함한다.
   - 리포트 본문은 `요약 → 변화 → 청구 특성 변화 → 운영 해석 → 다음 조치` 순서로 작성한다.
   - 리포트 상단에는 `월간 자동심사 현황 리포트` 타이틀과 기준월/비교월/데이터 확정 여부를 배치한다.
   - 시각화는 의미 전달에 유리한 경우에만 차트나 표로 추가한다.
   - 자동지급 현황 개요, 자동지급 유형 분석, 자동지급 제외유형 분석을 기본 섹션으로 둔다.

7. **자동 발송 및 보관**
   - 매월 1일 구독 계정 목록으로 자동 발송한다.
   - 생성된 리포트는 `생성됨`, `발송됨`, `미발송`, `재생성 필요` 상태로 리포트보관함에 누적 저장한다.
   - 사용자는 보관함에서 재조회 및 재다운로드할 수 있다.

## Data Flow Summary

`지급데이터 / 월말 스냅샷 / segment 변경 정보 / 비용환산 기준`
→ `월말 비교 및 변화 계산`
→ `변화 설명 / 성과 / 안정 상태 판정`
→ `운영모니터링 화면`
→ `월간 리포트 자동 생성`
→ `자동 발송`
→ `리포트보관함 저장`

## 운영 원칙

- 성과는 판단값이 아니라 ROI 보고용이다.
- 변화가 없어도 지표는 계속 노출하고, 상태는 `안정`으로 표시한다.
- 월말 스냅샷 기준이 기본이며, 조회기간 필터는 보조 기능이다.
- 청구유형 분류, 최종 지급 확정, 지급금액 단독 확정은 하지 않는다.
- 사람 심사가 필요한 건을 자동지급으로 강하게 밀어주는 흐름은 만들지 않는다.
